## Cross-Layer Priority Mapping

Different layers use different severity labels. This table maps them to universal priority tiers (P1-P5) for consistent sprint assignment in the remediation plan.

| Priority | Security | Functionality | Polish | Sprint |
|----------|----------|---------------|--------|--------|
| **P1** | CRITICAL | BROKEN | — | Sprint 1 (Blockers) |
| **P2** | HIGH | INCOMPLETE | REQUIRED_FOR_SHIP | Sprint 2 (High Priority) |
| **P3** | MEDIUM | MISSING, ORPHAN | SHOULD_HAVE | Sprint 3 (Improvements) |
| **P4** | LOW | FRAGILE, LOOSELY_COUPLED | NICE_TO_HAVE | Backlog |
| **P5** | — | — | FUTURE | Backlog (v2) |

When building the remediation plan, always sort by P-tier first, then by layer (Security before Functionality before Polish within the same tier).

## Scoring Formula

The Ship-Readiness Score (0-100) is calculated as:

```
Step 1: Check for CRITICAL ceiling
  IF any CRITICAL security finding exists → base score is capped at 50
  IF any BROKEN functionality exists → base score is capped at 60
  Take the LOWER cap if both apply.

Step 2: Deduct from the capped base (or from 100 if no cap)
  - Per HIGH security finding:         -10
  - Per MEDIUM security finding:       -3
  - Per LOW security finding:          -1
  - Per INCOMPLETE functionality:      -8
  - Per MISSING functionality:         -5
  - Per ORPHAN functionality:          -1
  - Per FRAGILE functionality:         -3
  - Per REQUIRED_FOR_SHIP polish:      -10
  - Per SHOULD_HAVE polish:            -2
  - Per NICE_TO_HAVE polish:           -0 (informational only)

Step 3: Apply floor
  Minimum score: 0

Example A: 1 CRITICAL, 2 HIGH, clean otherwise
  Cap: 50 (CRITICAL present) → 50 - 20 (2 HIGH) = 30 → NOT_READY

Example B: 0 CRITICAL, 3 HIGH, 2 INCOMPLETE
  No cap → 100 - 30 (3 HIGH) - 16 (2 INCOMPLETE) = 54 → CONDITIONAL

Example C: 0 CRITICAL, 1 MEDIUM, 2 SHOULD_HAVE
  No cap → 100 - 3 - 4 = 93 → SHIP_READY
```

Verdict Thresholds:
- 80-100: SHIP_READY
- 50-79:  CONDITIONAL (ship with documented known issues)
- 0-49:   NOT_READY (blockers must be resolved)

## Effort Estimation Guide

| Effort | Time Estimate | Description |
|--------|--------------|-------------|
| **S** (Small) | < 30 min | Config change, add a file, fix a one-liner, add a flag |
| **M** (Medium) | 30 min - 2 hours | Implement a missing handler, add auth to routes, write tests for a module |
| **L** (Large) | 2+ hours | Redesign auth flow, add comprehensive error handling, implement missing feature |
